# Especificación Funcional: Inicio de Sesión

**ID de Historia:** HU-001
**Versión:** 1.0
**Fecha:** 6 de mayo de 2026
**Estado:** Aprobado para implementación

> ⚠ **Estado actual (mayo 2026):** la implementación se desvió de esta spec. Este documento se conserva como referencia histórica de los requisitos originales. **Divergencias clave:**
>
> - **Google OAuth fue depreciado** (`../../../40db-backend/docs/auth.md §12`). El backend MVP usa Supabase Auth con **email + password**. La spec menciona Google como único proveedor; ya no aplica.
> - **No hay whitelist de dominios.** Cualquier correo puede registrarse vía signup en `LoginView.vue`.
> - **Signup explícito** desde la UI con tabs *Iniciar sesión* / *Crear cuenta*. La spec asumía registro automático tras OAuth.
> - **Onboarding obligatorio:** tras el primer signup el usuario es redirigido a `/onboarding` para completar `telefono` y `comuna_id` (`auth.md §7`). La spec no lo contemplaba.
> - **2FA** lo gestionará Supabase (no Google) cuando se habilite en producción.
> - **Logs de intentos fallidos**: Supabase Auth los registra internamente. El front no implementa logging propio.
>
> La descripción autoritativa del flujo actual vive en
> [`docs/integracion-backend/01-mapeo-contrato.md §1`](../../integracion-backend/01-mapeo-contrato.md).

## 1. Historia de Usuario

> **Como** usuario
> **quiero** iniciar sesión en el sistema
> **para** acceder a contenido personalizado.

---

## 2. Objetivo

Permitir que los usuarios autorizados accedan al sistema de forma segura mediante autenticación con Google OAuth, garantizando una experiencia fluida y conforme a los estándares de seguridad y accesibilidad definidos.

---

## 3. Alcance

### 3.1 Dentro del alcance

- Autenticación mediante Google OAuth como único proveedor.
- Registro automático de cuentas para usuarios cuyo correo pertenezca a dominios permitidos (whitelist).
- Gestión de sesiones persistentes con expiración por inactividad.
- Acceso a un dashboard personalizado tras la autenticación exitosa.
- Cierre de sesión manual desde la interfaz autenticada.
- Soporte para sesiones simultáneas en múltiples dispositivos.

### 3.2 Fuera del alcance

- Autenticación mediante usuario/contraseña local.
- Autenticación mediante otros proveedores OAuth (Microsoft, Apple, Facebook, etc.).
- Autenticación mediante magic link, SMS o PIN.
- Recuperación de contraseña (delegada a Google).
- Implementación de captcha.
- Verificación adicional de correo electrónico.
- Roles de usuario diferenciados (admin, editor, etc.).
- Localización en idiomas distintos al español.

---

## 4. Requisitos Funcionales

### 4.1 Autenticación

| ID | Requisito |
|------|------------|
| RF-01 | El sistema debe ofrecer un único método de autenticación: Google OAuth. |
| RF-02 | El botón "Iniciar sesión con Google" debe ser visible en la pantalla de login. |
| RF-03 | El sistema debe redirigir al usuario al flujo de autorización de Google al hacer clic en el botón. |
| RF-04 | Tras la autorización exitosa, el sistema debe recibir el token de Google y validar la identidad del usuario. |

### 4.2 Registro de cuenta (whitelist de dominios)

| ID | Requisito |
|------|------------|
| RF-05 | El sistema debe verificar que el dominio del correo del usuario pertenezca a la whitelist configurada. |
| RF-06 | Si el dominio está permitido y el usuario no existe, el sistema debe crear automáticamente la cuenta. |
| RF-07 | Si el dominio no está permitido, el sistema debe denegar el acceso y mostrar un mensaje de error genérico. |
| RF-08 | La administración de la whitelist de dominios debe definirse en una etapa técnica posterior (responsable y mecanismo de actualización). |

### 4.3 Gestión de sesión

| ID | Requisito |
|------|------------|
| RF-09 | La sesión debe permanecer activa hasta que el usuario cierre sesión manualmente. |
| RF-10 | La sesión debe expirar automáticamente tras 30 días de inactividad. |
| RF-11 | El sistema debe permitir sesiones simultáneas en múltiples dispositivos para un mismo usuario. |
| RF-12 | El sistema debe ofrecer un botón "Cerrar sesión" accesible desde el menú principal una vez autenticado. |

### 4.4 Doble factor de autenticación

| ID | Requisito |
|------|------------|
| RF-13 | El sistema no implementará un mecanismo propio de 2FA. |
| RF-14 | El 2FA será gestionado íntegramente por la cuenta de Google del usuario, según su propia configuración. |

### 4.5 Auditoría de intentos fallidos

| ID | Requisito |
|------|------------|
| RF-15 | El sistema debe registrar (log) todos los intentos fallidos de autenticación con fines de auditoría. |
| RF-16 | El registro debe incluir, como mínimo: timestamp, correo intentado (si está disponible), dirección IP y motivo del fallo. |
| RF-17 | El sistema no debe bloquear automáticamente al usuario ni a la IP tras intentos fallidos. |

### 4.6 Post-login

| ID | Requisito |
|------|------------|
| RF-18 | Tras un inicio de sesión exitoso, el sistema debe redirigir al usuario a su dashboard personal. |
| RF-19 | El dashboard debe mostrar información, configuraciones y datos asociados al perfil del usuario. |

### 4.7 Términos y condiciones

| ID | Requisito |
|------|------------|
| RF-20 | Debe mostrarse un aviso informativo junto al botón de Google con el siguiente texto (o equivalente): "Al iniciar sesión, aceptas nuestros Términos y Condiciones y nuestra Política de Privacidad". |
| RF-21 | Los enlaces a los Términos y a la Política de Privacidad deben estar disponibles desde el aviso. |
| RF-22 | No se requerirá un checkbox de aceptación explícita. |

### 4.8 Mensajes de error

| ID | Requisito |
|------|------------|
| RF-23 | Los mensajes de error deben ser genéricos por motivos de seguridad (por ejemplo, "No fue posible iniciar sesión. Verifica tus credenciales o contacta al administrador."). |
| RF-24 | No se debe revelar si el correo existe o si el problema corresponde a la whitelist. |

## 5. Requisitos No Funcionales

| ID | Categoría | Requisito |
|------|------------|------------|
| RNF-01 | Idioma | La interfaz debe estar disponible en español. |
| RNF-02 | Accesibilidad | La pantalla de login debe cumplir con los estándares WCAG 2.1 nivel AA. |
| RNF-03 | Seguridad | La comunicación con Google debe realizarse íntegramente sobre HTTPS. |
| RNF-04 | Seguridad | Los tokens de sesión deben almacenarse de forma segura (cookies HttpOnly y Secure, o equivalente). |
| RNF-05 | Seguridad | No se incluirán mecanismos propios de captcha; se confía en las protecciones nativas de Google. |
| RNF-06 | Auditoría | Los logs de intentos fallidos deben conservarse según la política de retención de la organización. |

---

## 6. Reglas de Negocio

| ID | Regla |
|------|--------|
| RN-01 | Solo los usuarios con correos pertenecientes a dominios en la whitelist pueden acceder al sistema. |
| RN-02 | La identidad del usuario se considera verificada por Google; no se requiere validación adicional del correo. |
| RN-03 | Existe un único tipo de usuario; no hay diferenciación de roles. |
| RN-04 | La sesión se renueva implícitamente con cada actividad del usuario; tras 30 días sin actividad, expira. |

---

## 7. Flujo Principal (Happy Path)

1. El usuario accede a la pantalla de login.
2. El usuario visualiza el botón "Iniciar sesión con Google" y el aviso informativo de términos.
3. El usuario hace clic en el botón.
4. El sistema redirige al flujo OAuth de Google.
5. El usuario se autentica en Google (incluyendo 2FA si lo tiene configurado).
6. Google redirige al sistema con el token de autorización.
7. El sistema valida el token y verifica el dominio del correo contra la whitelist.
8. **Si el dominio está permitido:**
   - Si el usuario no existe, se crea la cuenta automáticamente.
   - Se establece la sesión.
   - Se redirige al dashboard personal.
9. **Si el dominio no está permitido:**
   - Se registra el intento fallido.
   - Se muestra un mensaje de error genérico.
   - El usuario permanece en la pantalla de login.

---

## 8. Flujos Alternativos

### 8.1 Usuario cancela el flujo de Google

- El usuario es retornado a la pantalla de login sin mensaje de error.

### 8.2 Token de Google inválido o expirado

- El sistema registra el intento fallido.
- Se muestra un mensaje de error genérico.

### 8.3 Sesión expirada por inactividad

- Tras 30 días sin actividad, el sistema invalida la sesión.
- Al siguiente intento de acceso, el usuario es redirigido a la pantalla de login.

### 8.4 Cierre de sesión manual

- El usuario hace clic en "Cerrar sesión" desde el menú principal.
- El sistema invalida la sesión actual.
- El usuario es redirigido a la pantalla de login.

---

## 9. Criterios de Aceptación

- [ ] El usuario puede iniciar sesión con una cuenta de Google cuyo dominio esté en la whitelist.
- [ ] El usuario con un dominio no permitido recibe un mensaje de error genérico y no accede al sistema.
- [ ] El primer login con un dominio permitido crea automáticamente la cuenta del usuario.
- [ ] La sesión permanece activa indefinidamente mientras haya actividad.
- [ ] La sesión expira automáticamente tras 30 días de inactividad.
- [ ] El usuario puede tener múltiples sesiones activas en distintos dispositivos.
- [ ] El usuario puede cerrar sesión manualmente desde el menú principal.
- [ ] Los intentos fallidos se registran en el log de auditoría.
- [ ] El sistema no bloquea cuentas ni IPs tras intentos fallidos.
- [ ] El aviso informativo de términos y condiciones está visible junto al botón de Google.
- [ ] La pantalla de login cumple con WCAG 2.1 nivel AA.
- [ ] La interfaz se muestra correctamente en español.

1. Lista concreta de dominios permitidos (whitelist) y mecanismo de administración.
2. Política de retención de logs de auditoría.
3. Estrategia de almacenamiento de tokens (cookies, sessionStorage, etc.).
4. Diseño visual y wireframes de la pantalla de login y dashboard.
5. Texto definitivo de los Términos y Condiciones y la Política de Privacidad.
6. Endpoint(s) y configuración del cliente OAuth en Google Cloud Console.
