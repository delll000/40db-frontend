# Especificación: Monitoreo de Conectividad de Sensores IoT — Proyecto 40dB

## 1. Historia de usuario

> **Como funcionario, quiero monitorear la conectividad de los sensores IoT para asegurar la disponibilidad de los datos de medición.**

---

## 2. Contexto

El proyecto **40dB** contempla el despliegue de sensores de **ruido** para la medición ambiental. Esta especificación define un sistema de monitoreo de la conectividad de dichos sensores, con el fin de asegurar la disponibilidad de los datos de medición para los funcionarios que consumen los reportes de ruido.

---

## 3. Alcance del piloto

| Atributo                | Valor                                       |
| ----------------------- | ------------------------------------------- |
| Tipo de sensores        | Sensores de ruido (proyecto 40dB)           |
| Cantidad de sensores    | 1 a 2 sensores (primera instancia / piloto) |
| Distribución geográfica | Un único punto fijo dentro de una comuna    |
| Idioma de la interfaz   | Español                                     |

---

## 4. Roles y permisos

### 👨‍💼 Administrador

Equipo técnico responsable de la operación de la red de sensores ("nosotros").

### 👥 Funcionario

Usuario consumidor de los reportes de ruido. Necesita visibilidad básica sobre el estado operativo de los sensores.

### Matriz de permisos

| Funcionalidad                                         | 👥 Funcionario | 👨‍💼 Administrador |
| ----------------------------------------------------- | :------------: | :--------------: |
| Ver estado actual del sensor (funciona / no funciona) |       ✅       |        ✅        |
| Ver dashboard / mapa de monitoreo en tiempo real      |       ✅       |        ✅        |
| Ver histórico de eventos de conectividad (30 días)    |       ❌       |        ✅        |
| Configurar estado / parámetros del sensor             |       ❌       |        ✅        |
| Descargar reportes de disponibilidad en PDF           |       ❌       |        ✅        |
| Gestionar usuarios                                    |       ❌       |        ✅        |

---

## 5. Actores y precondiciones generales

**Actores:**

- **Administrador:** usuario autenticado con cuenta dedicada.
- **Funcionario:** usuario con acceso limitado a la vista simplificada de monitoreo.
- **Sensor IoT (40dB):** dispositivo que emite reportes periódicos al backend.
- **Sistema de monitoreo:** plataforma web que centraliza estado, históricos, alertas y reportes.

**Precondiciones generales:**

- Existe al menos un sensor 40dB instalado, encendido y configurado para reportar al backend.
- El backend recibe y almacena los reportes de los sensores.
- Existe al menos un usuario administrador creado en el sistema.

---

## 6. Happy paths

### 🟢 HP-1 — Administrador revisa el estado de los sensores

**Actor:** Administrador
**Objetivo:** verificar en tiempo real que los sensores están operativos.

**Flujo:**

1. El administrador ingresa a la URL del sistema desde su navegador.
2. Ingresa usuario y contraseña.
3. El sistema valida las credenciales y lo redirige al **dashboard**.
4. El dashboard muestra:
   - Mapa con la ubicación del sensor.
   - Tabla con ID, ubicación, último reporte, estado, batería y señal.
5. El administrador identifica que todos los sensores están **en línea**.
6. El administrador cierra sesión.

**Resultado esperado:** el administrador confirma la disponibilidad de los sensores en menos de 1 minuto desde el ingreso.

---

### 🟢 HP-2 — Funcionario consulta si el sensor está funcionando

**Actor:** Funcionario
**Objetivo:** confirmar que los datos de medición están disponibles antes de consultar reportes de ruido.

**Flujo:**

1. El funcionario accede al sistema desde un navegador (escritorio o móvil).
2. Visualiza la vista simplificada con la ubicación y el estado actual del sensor.
3. Ve un indicador claro: **funcionando** ✅ o **no funcionando** ❌.
4. Procede con confianza a consultar los reportes de ruido en su sistema habitual.

**Resultado esperado:** el funcionario obtiene en una sola vista una respuesta inequívoca sobre el estado del sensor.

---

### 🟢 HP-3 — Sensor pasa a "fuera de línea" y se notifica al administrador

**Actor:** Sistema + Administrador
**Objetivo:** detectar y comunicar oportunamente la desconexión de un sensor.

**Flujo:**

1. El sensor deja de reportar datos al backend.
2. Transcurridos **15 minutos** sin reportes, el sistema cambia el estado del sensor a **fuera de línea**.
3. El sistema envía una **notificación por correo electrónico** al o los administradores configurados.
4. El estado se refleja en el dashboard en el siguiente ciclo de actualización (≤ 5 min).
5. El administrador recibe el correo, ingresa al sistema y verifica el evento en el histórico.

**Resultado esperado:** el administrador toma conocimiento de la desconexión a más tardar 20 minutos después de que el sensor dejó de reportar.

---

### 🟢 HP-4 — Administrador descarga reporte de disponibilidad

**Actor:** Administrador
**Objetivo:** obtener un reporte PDF de disponibilidad para análisis o entrega.

**Flujo:**

1. El administrador inicia sesión.
2. Navega al módulo de **Reportes**.
3. Selecciona el rango de fechas (dentro de los últimos 30 días) y el sensor.
4. Hace clic en **"Generar reporte"**.
5. El sistema genera un PDF con los datos de disponibilidad del periodo solicitado.
6. El administrador descarga el archivo.

**Resultado esperado:** PDF descargado correctamente con el detalle de disponibilidad del periodo.

---

### 🟢 HP-5 — Administrador consulta histórico de conectividad

**Actor:** Administrador
**Objetivo:** revisar eventos de conectividad recientes para diagnóstico.

**Flujo:**

1. El administrador inicia sesión.
2. Selecciona un sensor desde el dashboard.
3. Accede a la vista de **histórico de conectividad**.
4. Aplica filtros (por estado, por rango de fechas dentro de 30 días).
5. Visualiza la línea de tiempo de eventos (en línea / intermitente / fuera de línea).

**Resultado esperado:** el administrador identifica patrones o eventos relevantes para tomar decisiones operativas.

## 7. Flujos alternativos y de error

### 🔴 AF-1 — Credenciales inválidas

- **Trigger:** usuario o contraseña incorrectos.
- **Comportamiento:** el sistema muestra un mensaje genérico ("Usuario o contraseña incorrectos") sin revelar cuál de los dos campos falló.
- **Política:** tras 5 intentos fallidos consecutivos, se bloquea el acceso temporalmente por 15 minutos.

### 🔴 AF-2 — Sensor en estado intermitente

- **Trigger:** el sensor reporta de forma irregular (con interrupciones menores a 15 minutos pero recurrentes).
- **Comportamiento:** el sistema marca el sensor como **intermitente** en el dashboard.
- **Notificación:** no se envía correo (solo se notifica el paso a "fuera de línea"), pero el evento queda registrado en el histórico.

### 🔴 AF-3 — Sensor recupera la conectividad

- **Trigger:** un sensor en estado "fuera de línea" vuelve a reportar.
- **Comportamiento:** el sistema cambia el estado a **en línea** en el siguiente ciclo de actualización y registra el evento en el histórico.
- **Notificación:** opcional (a definir si se notifica también la recuperación).

### 🔴 AF-4 — Backend no recibe datos de ningún sensor

- **Trigger:** caída del backend o de la red de comunicación.
- **Comportamiento:** todos los sensores aparecerán como "fuera de línea" tras 15 minutos. Se recomienda contar con una alerta de salud del propio backend (fuera del alcance de esta versión, a considerar en evoluciones).

### 🔴 AF-5 — Funcionario intenta acceder a una funcionalidad de admin

- **Trigger:** el funcionario intenta acceder por URL directa a históricos, configuración o reportes.
- **Comportamiento:** el sistema responde con un error 403 (Forbidden) o redirige a la vista permitida con un mensaje informativo.

### 🔴 AF-6 — Generación de reporte sin datos

- **Trigger:** el administrador solicita un reporte para un rango de fechas sin eventos.
- **Comportamiento:** el PDF se genera igualmente, indicando explícitamente "Sin eventos registrados en el periodo".

## 8. Criterios de aceptación

### CA-1 — Visualización del estado

- [ ] El dashboard muestra el mapa con la ubicación de los sensores.
- [ ] El dashboard muestra una tabla con los campos: ID, ubicación, último reporte, estado, batería, señal.
- [ ] La información se actualiza automáticamente cada **5 minutos**.
- [ ] Los tres estados (en línea, intermitente, fuera de línea) se distinguen visualmente (ej. colores: verde, amarillo, rojo).

### CA-2 — Detección de desconexión

- [ ] Cuando un sensor no reporta durante **15 minutos**, su estado cambia a "fuera de línea".
- [ ] El cambio de estado se refleja en el dashboard en el siguiente ciclo de actualización.

### CA-3 — Alertas

- [ ] El sistema envía un correo electrónico al/los administrador(es) cuando un sensor pasa a "fuera de línea".
- [ ] El correo contiene: ID del sensor, ubicación, hora del último reporte y enlace al dashboard.

### CA-4 — Roles y permisos

- [ ] El administrador accede a todas las funcionalidades.
- [ ] El funcionario solo accede a la vista simplificada del estado actual.
- [ ] Los intentos de acceso no autorizados son rechazados.

### CA-5 — Histórico

- [ ] El administrador puede consultar eventos de conectividad de los últimos **30 días**.
- [ ] Los eventos pueden filtrarse por estado y rango de fechas.

### CA-6 — Reportes

- [ ] El administrador puede generar y descargar reportes de disponibilidad en **PDF**.
- [ ] El reporte incluye el rango de fechas solicitado y el detalle de eventos.

### CA-7 — Filtros del dashboard

- [ ] Disponibles filtros por: zona geográfica, tipo de sensor y estado.

### CA-8 — Acceso multiplataforma

- [ ] El sistema es **responsive** y se visualiza correctamente en escritorio y móvil.

### CA-9 — Autenticación

- [ ] El acceso de administradores requiere usuario y contraseña.
- [ ] La sesión se cierra automáticamente tras un período de inactividad (sugerido: 30 min).

---

## 9. Reglas de negocio

| ID   | Regla                                                                                                                                     |
| ---- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| RN-1 | Un sensor sin reportes durante ≥ 15 min se considera **fuera de línea**.                                                                  |
| RN-2 | Un sensor con interrupciones < 15 min pero recurrentes se considera **intermitente** (definición operativa: 3+ interrupciones en 1 hora). |
| RN-3 | El histórico se conserva por **30 días corridos**; los datos más antiguos son depurados o archivados.                                     |
| RN-4 | Solo los administradores pueden generar reportes y consultar históricos.                                                                  |
| RN-5 | Las alertas por correo se envían **únicamente** al pasar a estado "fuera de línea" (no en intermitencias).                                |
| RN-6 | El dashboard se actualiza cada 5 minutos.                                                                                                 |

## 10. Funcionalidades detalladas

### 10.1. Monitoreo en tiempo real

- Dashboard con **mapa geográfico** + **listado tabular**.
- **Frecuencia de actualización:** cada 5 minutos.
- **Información mostrada por sensor:** ID, ubicación, último reporte, estado, batería, señal.

### 10.2. Estados de conectividad

1. **En línea** — el sensor reporta normalmente.
2. **Intermitente** — el sensor reporta de forma irregular.
3. **Fuera de línea** — el sensor no ha reportado en los últimos 15 minutos.

### 10.3. Filtros disponibles

- Por zona geográfica.
- Por tipo de sensor.
- Por estado de conectividad.

### 10.4. Alertas

- Notificaciones por correo al pasar a "fuera de línea".
- Sin acciones correctivas automáticas (solo notificación).
