# Especificación Funcional: Modificación de Estado de Alertas de Ruido

> ⚠ **Estado actual (mayo 2026):** este documento usa la nomenclatura *Alerta* que fue **unificada con `Reporte`** durante la integración con el backend (decisión F6, ver
> [`docs/integracion-backend/README.md`](../../integracion-backend/README.md)). **Divergencias clave:**
>
> - **El concepto `Alerta` no existe en el backend.** Lo que la spec llama "alerta" es ahora un `Reporte` (`../../../40db-backend/docs/bbdd.md §3.6`). Funcionarios gestionan reportes ciudadanos, no alertas auto-generadas por sensores. El umbral de 40 dB ya no dispara registros automáticos.
> - **Los estados cambian:** `activa / atendida / cerrada` → `En espera / En atencion / Atendido / Descartado` (`api.md §4.13`).
> - **No hay reversión de 5 minutos.** Un reporte no puede volver a un estado anterior una vez cambiado. La trazabilidad vive en `historial_estado` (append-only, `bbdd.md §3.7`).
> - **Estados terminales:** `Atendido` y `Descartado` no aceptan más transiciones (regla del back, `api.md §4.9`).
> - **Comentario obligatorio solo en `Descartado`** — no en cada cambio de estado como pedía la spec. Para `En atencion` y `Atendido` es opcional.
> - **Endpoint:** `PATCH /api/v1/reportes/{id}/estado` con body `{ nuevo_estado, comentario? }`.
> - **Side effect server-side:** la transición `En espera → En atencion` setea automáticamente `atendido_por_id = auth.uid` (`api.md §4.9`).
>
> La descripción autoritativa del flujo actual vive en
> [`docs/integracion-backend/01-mapeo-contrato.md §3`](../../integracion-backend/01-mapeo-contrato.md).
> La implementación está en `src/views/funcionario/FuncionarioHomeView.vue`.

## 1. Historia de Usuario

> **Como** funcionario municipal,
> **quiero** modificar el estado de las alertas detectadas por los sensores de ruido,
> **para** registrar la trazabilidad de la respuesta municipal frente a eventos de ruido ambiental.

---

## 2. Contexto

El sistema corresponde a una **plataforma de monitoreo de ruido ambiental** desplegada para municipalidades. Las alertas se generan automáticamente a partir de **sensores IoT** instalados en zonas de interés municipal, cuando se detectan niveles de ruido sobre el umbral de referencia de **40 dB**.

Las alertas son gestionadas internamente por funcionarios municipales para dar seguimiento operativo y dejar registro de la respuesta del municipio frente a cada evento detectado.

---

## 3. Alcance

### Dentro del alcance

- Modificación manual del estado de una alerta por parte de un funcionario autenticado.
- Registro automático de trazabilidad (historial) de cada cambio de estado.
- Visualización del historial de cambios dentro del detalle de la alerta.
- Reversión del estado dentro de una ventana de tiempo definida.
- Acceso desde panel web responsive (escritorio y móvil).

### Fuera del alcance (versión actual)

- Notificaciones a vecinos o ciudadanos (contemplado a futuro).
- Aplicaciones móviles nativas (iOS / Android).
- Adjuntar archivos o evidencia al cambio de estado.
- Edición masiva de múltiples alertas.
- Roles diferenciados con permisos distintos sobre el cambio de estado.

---

## 4. Actores

| Actor                      | Descripción                                                                                                                 |
| -------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| **Funcionario**            | Usuario municipal autenticado en la plataforma. Es el único rol con permisos para modificar el estado de las alertas.       |
| **Sistema (sensores IoT)** | Genera las alertas automáticamente cuando se supera el umbral de 40 dB. No interviene en la modificación manual del estado. |

---

## 5. Estados de la Alerta

El ciclo de vida de una alerta contempla tres estados:

| Estado       | Descripción                                                                                         |
| ------------ | --------------------------------------------------------------------------------------------------- |
| **Activa**   | Estado inicial de toda alerta generada por el sensor. Aún no ha sido gestionada por un funcionario. |
| **Atendida** | El funcionario ha tomado conocimiento o acción sobre la alerta.                                     |
| **Cerrada**  | La alerta ha sido finalizada y no requiere más gestión.                                             |

### Diagrama de transiciones

```
        ┌─────────┐         ┌──────────┐         ┌─────────┐
        │ Activa  │  ────►  │ Atendida │  ────►  │ Cerrada │
        └─────────┘         └──────────┘         └─────────┘
             │                    │                    │
             │                    │                    │
             └─── reversión dentro de 5 min ──────────┘
                       (al estado anterior)
```

### Reglas de transición

- **Activa → Atendida**: permitida.
- **Atendida → Cerrada**: permitida.
- **Activa → Cerrada**: no permitida directamente; debe pasar primero por "Atendida".
- **Reversión**: cualquier cambio puede ser revertido al estado inmediatamente anterior dentro de los **5 minutos** posteriores al cambio.

## 6. Requisitos Funcionales

### RF-01 — Autenticación obligatoria

Solo los usuarios autenticados con rol "Funcionario" pueden modificar el estado de una alerta.

### RF-02 — Modificación de estado

El funcionario podrá cambiar el estado de una alerta desde el detalle de la misma, seleccionando el nuevo estado válido según las reglas de transición.

### RF-03 — Comentario obligatorio

Al modificar el estado, el funcionario **debe** ingresar un comentario en texto libre que justifique el cambio. El sistema no permitirá guardar el cambio sin un comentario.

### RF-04 — Registro de trazabilidad

Cada cambio de estado generará un registro automático en el historial de la alerta con los siguientes datos:

- Fecha y hora del cambio (timestamp).
- Identificación del funcionario que realizó el cambio.
- Estado anterior.
- Estado nuevo.
- Comentario ingresado.

### RF-05 — Visualización del historial

El historial de cambios estará disponible dentro del detalle de cada alerta, ordenado cronológicamente (más reciente primero).

### RF-06 — Reversión dentro de ventana temporal

El funcionario podrá revertir un cambio al estado anterior **únicamente dentro de los 5 minutos posteriores** al cambio. Pasado ese tiempo, el cambio se considera definitivo y solo podrá registrarse un nuevo cambio.

### RF-07 — Modificación individual

Solo se permite modificar el estado de **una alerta a la vez**. No existe edición masiva.

### RF-08 — Acceso responsive

La funcionalidad estará disponible en un panel web responsive, accesible tanto desde escritorio como desde dispositivos móviles vía navegador.

### RF-09 — Sin notificaciones externas

La modificación de estado **no generará notificaciones** a actores externos (vecinos, ciudadanos). Es una operación interna del municipio.

---

## 7. Requisitos No Funcionales

| Código | Requisito                                                                                                                          |
| ------ | ---------------------------------------------------------------------------------------------------------------------------------- |
| RNF-01 | El historial de cambios es **inmutable** (salvo por la ventana de reversión de 5 minutos).                                         |
| RNF-02 | La interfaz debe ser **responsive** y funcionar correctamente en navegadores modernos de escritorio y móviles.                     |
| RNF-03 | El cambio de estado debe quedar registrado de forma **transaccional**: si falla el registro del historial, no se aplica el cambio. |
| RNF-04 | El sistema debe registrar el timestamp del cambio en zona horaria del municipio, con precisión al segundo.                         |

## 8. Criterios de Aceptación

### CA-01 — Cambio de estado exitoso

**Dado** que soy un funcionario autenticado,
**cuando** abro el detalle de una alerta en estado "Activa" y la cambio a "Atendida" ingresando un comentario,
**entonces** el sistema actualiza el estado, registra la entrada en el historial con todos los datos, y la alerta queda visible como "Atendida".

### CA-02 — Comentario obligatorio

**Dado** que estoy modificando el estado de una alerta,
**cuando** intento guardar el cambio sin ingresar un comentario,
**entonces** el sistema impide la acción y muestra un mensaje indicando que el comentario es obligatorio.

### CA-03 — Transición inválida

**Dado** que una alerta se encuentra en estado "Activa",
**cuando** intento cambiarla directamente a "Cerrada",
**entonces** el sistema no permite la transición y muestra solo las transiciones válidas ("Atendida").

### CA-04 — Reversión dentro del tiempo permitido

**Dado** que cambié el estado de una alerta hace menos de 5 minutos,
**cuando** ejecuto la acción de revertir,
**entonces** el estado vuelve al anterior y se registra una nueva entrada en el historial indicando la reversión.

### CA-05 — Reversión fuera del tiempo permitido

**Dado** que el cambio de estado se realizó hace más de 5 minutos,
**cuando** intento revertirlo,
**entonces** el sistema no permite la reversión y solo ofrece registrar un nuevo cambio de estado.

### CA-06 — Visualización del historial

**Dado** que abro el detalle de una alerta con varios cambios de estado,
**cuando** consulto el historial,
**entonces** veo todos los cambios ordenados del más reciente al más antiguo, con fecha, funcionario, estado anterior, estado nuevo y comentario.

### CA-07 — Acceso restringido a no autenticados

**Dado** que no he iniciado sesión,
**cuando** intento acceder a la funcionalidad de modificación de estado,
**entonces** el sistema me redirige al login y no permite la operación.

### CA-08 — Acceso responsive

**Dado** que accedo a la plataforma desde el navegador de un teléfono móvil,
**cuando** abro el detalle de una alerta,
**entonces** puedo modificar su estado con la misma funcionalidad que en escritorio.

## 9. Modelo de Datos (referencial)

### Entidad: `Alerta`

| Campo             | Tipo      | Descripción                  |
| ----------------- | --------- | ---------------------------- |
| `id`              | UUID      | Identificador único          |
| `sensor_id`       | UUID      | Sensor que originó la alerta |
| `nivel_db`        | decimal   | Nivel de ruido detectado     |
| `fecha_deteccion` | timestamp | Momento de generación        |
| `estado_actual`   | enum      | Activa / Atendida / Cerrada  |
| `ubicacion`       | geo       | Coordenadas del sensor       |

### Entidad: `HistorialEstadoAlerta`

| Campo             | Tipo      | Descripción                                       |
| ----------------- | --------- | ------------------------------------------------- |
| `id`              | UUID      | Identificador único del registro                  |
| `alerta_id`       | UUID      | Referencia a la alerta                            |
| `funcionario_id`  | UUID      | Funcionario que ejecutó el cambio                 |
| `estado_anterior` | enum      | Estado previo al cambio                           |
| `estado_nuevo`    | enum      | Estado posterior al cambio                        |
| `comentario`      | text      | Justificación del cambio                          |
| `fecha_cambio`    | timestamp | Momento del cambio                                |
| `es_reversion`    | boolean   | Indica si el registro corresponde a una reversión |

---

## 10. Supuestos y Dependencias

- Existe un módulo previo de **autenticación** que provee la sesión del funcionario y su identificador.
- Existe un módulo previo de **detección de alertas** que genera registros en estado "Activa" automáticamente desde los sensores IoT.
- Existe una vista de **listado y detalle de alertas** donde se integra esta funcionalidad.
- La base de datos del sistema soporta operaciones transaccionales.
