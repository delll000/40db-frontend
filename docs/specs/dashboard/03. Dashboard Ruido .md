# Especificación — HU-01: Dashboard de Ruido en Tiempo Real

> ⚠ **Estado actual (mayo 2026):** la implementación cambió varios detalles del comportamiento del heatmap. **Divergencias clave:**
>
> - **Filtros:** la spec lista `rango de fechas / horario / zona / sensor`. El back agrega por **bbox del viewport + ventana temporal + bucket** (`../../../40db-backend/docs/api.md §4.3`). No hay filtros por "zona" ni "sensor" individual — el heatmap es agregado por celda geo (~100 m).
> - **Sensores IoT activos** en KPI: deriva del `total_cells` de la metadata, no de un catálogo (el back no expone listado de sensores; ver
>   [`02-endpoints-faltantes-back.md §1`](../../integracion-backend/02-endpoints-faltantes-back.md)).
> - **Zona más ruidosa:** se etiqueta con coords (`-33.4372, -70.6483`) porque el back no nombra celdas.
> - **Alerta sostenida (>75 dB > 10 min):** pendiente
>   [`02-endpoints-faltantes-back.md §5`](../../integracion-backend/02-endpoints-faltantes-back.md). El KPI queda en `—` hasta que el back lo provea.
> - **Capa normativa DS 14/2024:** no implementada por ahora.
> - **Acceso `/heatmap` standalone** está gateado por `VITE_ENABLE_ADMIN_DEMO` (rama 6). Para vecinos el heatmap vive **embebido** en `VecinoHomeView.vue`.
> - **Google institucional**: depreciado — ver el banner de `04. Especificacion Inicio Sesion.md`.
> - **Exportes** PNG / CSV / PDF se mantienen (vía `jspdf` + `html2canvas`, client-side).
>
> La descripción autoritativa vive en
> [`docs/integracion-backend/01-mapeo-contrato.md §6`](../../integracion-backend/01-mapeo-contrato.md).

> **Como** funcionario, **quiero** visualizar los indicadores clave y el mapa de calor **para** evaluar el impacto del ruido en la comuna en tiempo real.

## 1. Vista por defecto

Al ingresar al dashboard se muestra el mapa de calor centrado automáticamente en la comuna de Maipú, con datos de las **últimas 24 horas**.

## 2. Indicadores Clave (KPIs)

Se muestran 4 KPIs en el panel superior, siempre visibles y actualizados en tiempo real:

| KPI                            | Descripción                                                                     |
| ------------------------------ | ------------------------------------------------------------------------------- |
| **dB promedio comunal actual** | Promedio aritmético de la última lectura válida de cada sensor activo.          |
| **Zonas sobre la norma**       | Cantidad de zonas que superan el umbral del DS 14/2024 según horario aplicable. |
| **Sensores IoT activos**       | Sensores con lectura reciente (últimos 15 min).                                 |
| **Zona más ruidosa**           | Sector de la comuna con mayor nivel de dB en la última ventana de 5 min.        |

> En esta entrega **no se contemplan reportes ciudadanos**: el origen único de datos son los sensores IoT.

## 3. Tiempo real

El dashboard se refresca automáticamente cada **60 segundos** sin necesidad de recargar la página.

## 4. Mapa de calor

- **Granularidad:** interpolación por celdas de ~100×100 m, con zoom hasta nivel de calle.
- **Escala cromática:**
  - 🟢 Verde: < 55 dB
  - 🟡 Amarillo: 55–65 dB
  - 🟠 Naranja: 65–75 dB
  - 🔴 Rojo: > 75 dB
- **Zonas sin datos:** se representan en gris (no se interpolan).
- **Ponderación:** todos los sensores tienen el mismo peso; el heatmap se genera por interpolación simple entre sus lecturas.
- **Capa normativa (opcional):** destaca las zonas donde se supera el límite del DS 14/2024 según horario diurno o nocturno.

## 5. Filtros

| Filtro                       | Comportamiento                                             |
| ---------------------------- | ---------------------------------------------------------- |
| **Rango de fechas**          | Date picker (desde/hasta). Por defecto: últimas 24 h.      |
| **Rango horario**            | Restringe a una franja horaria dentro del rango de fechas. |
| **Zona / sector**            | Acota mapa y KPIs a un sector específico de Maipú.         |
| **ID / ubicación de sensor** | Aísla la visualización a un sensor específico.             |

## 6. Alertas visuales

Cuando una zona supera **>75 dB sostenidos por más de 10 minutos**:

- Aparece un indicador parpadeante en el mapa sobre la zona afectada.
- Se muestra un aviso en el panel de KPIs.

## 7. Comparación histórica

El funcionario puede comparar el estado actual con el **promedio histórico de la misma franja horaria**.

## 8. Acceso

Solo funcionarios autenticados con cuenta de **Google institucional (rol "municipalidad")** pueden acceder al dashboard.

## 9. Dispositivos

El dashboard es **totalmente responsive**: funciona en escritorio, tablet y móvil, permitiendo consultarlo tanto en oficina como en terreno.

## 10. Idioma y unidades

- Interfaz en **español de Chile**.
- Decibeles en escala **dB(A)**.

## 11. Exportación

- Vista actual del heatmap → **PNG**.
- Datos filtrados → **CSV** o **PDF**.

## Happy Path

Flujo principal exitoso del funcionario al consultar el dashboard:

1. El funcionario accede a la URL del dashboard desde su navegador (escritorio, tablet o móvil).
2. El sistema detecta que no hay sesión activa y lo redirige al login con Google institucional.
3. El funcionario se autentica con su cuenta de Google Workspace (rol "municipalidad").
4. El sistema valida el rol y carga el dashboard.
5. Se muestra automáticamente:
   - El **mapa de calor** centrado en Maipú con datos de las últimas 24 h.
   - Los **4 KPIs** (dB promedio, zonas sobre la norma, sensores activos, zona más ruidosa).
6. El funcionario interactúa con el mapa (zoom hasta calle, hover sobre celdas para ver dB exacto).
7. El funcionario aplica filtros (rango de fechas, rango horario, zona/sector o sensor específico).
8. El sistema actualiza el mapa y los KPIs con los datos filtrados.
9. El funcionario activa la capa normativa para ver qué zonas superan el DS 14/2024.
10. Cada 60 segundos el dashboard se refresca automáticamente con las últimas lecturas.
11. El funcionario exporta la vista actual como PNG y/o los datos filtrados como CSV/PDF.
12. El funcionario cierra sesión.

## Flujos Alternativos / Errores

Casos en los que el flujo principal se desvía o falla:

| ID        | Escenario                                                                  | Comportamiento esperado                                                                                                                                    |
| --------- | -------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **FA-01** | Usuario sin rol "municipalidad" intenta acceder.                           | El sistema bloquea el ingreso y muestra mensaje "No tienes permisos para acceder a este dashboard".                                                        |
| **FA-02** | Usuario no autenticado accede a la URL.                                    | Redirección automática al login con Google institucional.                                                                                                  |
| **FA-03** | No hay sensores activos en este momento.                                   | El mapa se muestra completamente en gris ("sin datos"); el KPI "Sensores activos" indica 0 y se muestra un banner: "No hay datos de sensores disponibles". |
| **FA-04** | Solo algunos sensores reportan datos.                                      | Las zonas sin lecturas recientes se muestran en gris; los KPIs se calculan sólo con los sensores activos.                                                  |
| **FA-05** | El backend no responde al cargar el dashboard.                             | Se muestra un mensaje de error con botón "Reintentar"; los KPIs y el mapa quedan en estado "cargando".                                                     |
| **FA-06** | Falla la actualización automática (cada 60 s).                             | El dashboard mantiene los últimos datos cargados y muestra un indicador discreto: "Última actualización: HH:MM — Reintentando…".                           |
| **FA-07** | El filtro aplicado no devuelve resultados (ej: rango de fechas sin datos). | Se muestra el mapa en gris con el mensaje: "No hay datos para los filtros seleccionados"; los KPIs muestran "—".                                           |
| **FA-08** | Sensor con valor anómalo (ej: dB negativos o > 200).                       | El sistema descarta la lectura inválida y la excluye del cálculo de KPIs y del heatmap.                                                                    |
| **FA-09** | Pérdida de conexión del cliente (sin internet).                            | El dashboard mantiene la última vista cargada y muestra un banner: "Sin conexión — los datos pueden estar desactualizados".                                |
| **FA-10** | Falla en la exportación (PNG / CSV / PDF).                                 | Se muestra mensaje de error: "No se pudo generar el archivo. Inténtalo nuevamente".                                                                        |
| **FA-11** | El usuario accede desde un navegador no compatible.                        | Se muestra un mensaje recomendando navegadores soportados (Chrome, Edge, Firefox, Safari en versiones recientes).                                          |
| **FA-12** | La sesión del funcionario expira mientras usa el dashboard.                | Se muestra un modal: "Tu sesión ha expirado. Vuelve a iniciar sesión" y se redirige al login.                                                              |

## Supuestos y dependencias

- Existe una flota de sensores IoT desplegada en la comuna de Maipú con conectividad activa.
- Existe un backend que recibe, valida y almacena las lecturas con su timestamp y geolocalización.
- Existe un servicio de autenticación con Google Workspace institucional configurado para el rol "municipalidad".
- Los sectores oficiales de la comuna están definidos y disponibles como capa geográfica.
