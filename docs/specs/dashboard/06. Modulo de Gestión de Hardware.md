## 1. Historia de usuario

> Como administrador del sistema, quiero contar con un módulo en `/admin/hardware` que me permita registrar nuevos dispositivos IoT ingresando su ID técnico y coordenadas geográficas, además de editar o dar de baja equipos existentes, para que la base de datos de 40dB mantenga un inventario físico actualizado que alimente correctamente al mapa de calor y al dashboard de monitoreo.

---

## 2. Objetivo

Proveer una interfaz administrativa que permita gestionar el ciclo de vida (alta, modificación, baja lógica) del inventario físico de dispositivos IoT, garantizando que las fuentes de datos visuales (mapa de calor y dashboard de monitoreo) reciban información consistente y actualizada.

---

## 3. Alcance

### Dentro del alcance

- Listado, búsqueda y filtrado de dispositivos IoT.
- Registro de nuevos dispositivos (ID técnico + coordenadas).
- Edición de dispositivos existentes.
- Baja lógica (soft delete) de dispositivos.
- Validaciones de integridad y formato.
- Sincronización con mapa de calor y dashboard al refrescar.

### Fuera del alcance (v1)

- Carga masiva por CSV/Excel.
- Edición o baja masiva.
- Auditoría visible (historial de cambios) en la UI.
- Exportación de datos.
- Previsualización en mapa interactivo durante el registro.
- Sincronización en tiempo real (websockets).
- Campos adicionales del dispositivo (nombre, tipo, descripción).

| Rol                | Acceso a `/admin/hardware`       |
| ------------------ | -------------------------------- |
| Administrador      | Acceso total (lectura/escritura) |

## 5. Modelo de datos

### Entidad: `Device`

| Campo          | Tipo         | Restricciones                  | Editable        |
| -------------- | ------------ | ------------------------------ | --------------- |
| `id`           | UUID         | PK, autogenerado               | No              |
| `technical_id` | string       | Único, requerido               | Sí              |
| `latitude`     | decimal(9,6) | Rango `[-90, 90]`, requerido   | Sí              |
| `longitude`    | decimal(9,6) | Rango `[-180, 180]`, requerido | Sí              |
| `status`       | enum         | `active` \| `decommissioned`   | Sí (vía acción) |
| `created_at`   | timestamp    | Autogestionado                 | No              |
| `updated_at`   | timestamp    | Autogestionado                 | No              |

## 6. Interfaz de usuario

### 6.1 Vista principal — Listado

- Tabla paginada (25 registros por página).
- Columnas: **ID técnico**, **Latitud**, **Longitud**, **Estado**, **Fecha de registro**, **Acciones**.
- Barra de búsqueda por **ID técnico**.
- Filtro por **Estado**: Todos / Activos / Dados de baja.
- Botón principal **"Registrar dispositivo"**.

### 6.2 Formulario — Registrar / Editar

- Campo **ID técnico** (texto, único).
- Campo **Latitud** (numérico, 6 decimales, rango [-90, 90]).
- Campo **Longitud** (numérico, 6 decimales, rango [-180, 180]).
- Botones: **Guardar** / **Cancelar**.

### 6.3 Acciones por fila

- ✏️ **Editar** → abre formulario precargado.
- 🗑️ **Dar de baja** → abre modal de confirmación.

### 6.4 Diseño responsivo

- **Desktop**: experiencia principal y prioritaria.
- **Tablet y móvil**: completamente responsive y funcional (todas las operaciones disponibles: registrar, editar, dar de baja).

### 6.5 Idioma

- Toda la UI en **español**: etiquetas, mensajes, validaciones, notificaciones.

## 7. Happy paths

### 7.1 Registrar un nuevo dispositivo

1. Administrador navega a `/admin/hardware`.
2. Hace clic en **"Registrar dispositivo"**.
3. Completa los campos: ID técnico, latitud, longitud.
4. Hace clic en **Guardar**.
5. El sistema valida formato, rangos y unicidad del ID técnico.
6. El dispositivo se crea con estado `active` y `created_at` actual.
7. Se muestra **toast de éxito**: _"Dispositivo registrado correctamente"_.
8. La tabla se actualiza mostrando el nuevo registro.

### 7.2 Editar un dispositivo existente

1. Administrador localiza el dispositivo (búsqueda o navegación).
2. Hace clic en **Editar** en la fila correspondiente.
3. Se abre el formulario con los datos actuales precargados.
4. Modifica ID técnico y/o coordenadas.
5. Hace clic en **Guardar**.
6. El sistema valida formato, rangos y unicidad.
7. El registro se actualiza y se actualiza `updated_at`.
8. Se muestra **toast de éxito**: _"Dispositivo actualizado correctamente"_.
9. La tabla refleja los cambios.

### 7.3 Dar de baja un dispositivo

1. Administrador localiza el dispositivo activo.
2. Hace clic en **Dar de baja**.
3. Se abre modal de confirmación: _"¿Seguro que deseas dar de baja este dispositivo?"_.
4. Hace clic en **Confirmar**.
5. El sistema cambia el estado a `decommissioned` (sin eliminar el registro).
6. Se muestra **toast de éxito**: _"Dispositivo dado de baja correctamente"_.
7. El dispositivo deja de aparecer en mapa de calor y dashboard tras el siguiente refresh.

### 7.4 Buscar y filtrar

1. Administrador escribe parte del ID técnico en la barra de búsqueda.
2. La tabla filtra resultados al instante (o al presionar enter).
3. Combina con filtro de estado (Activos / Dados de baja / Todos).
4. La tabla muestra el subconjunto correspondiente.

## 8. Caminos alternativos / errores

### 8.1 Acceso no autorizado

- Usuario sin rol administrador accede a `/admin/hardware` → respuesta **HTTP 403** + pantalla de acceso denegado.

### 8.2 ID técnico duplicado

- Al guardar, si ya existe otro dispositivo con el mismo ID técnico → mensaje inline: _"Ya existe un dispositivo con este ID técnico"_. No se guarda.

### 8.3 Coordenadas inválidas

- Latitud fuera de `[-90, 90]` → _"La latitud debe estar entre -90 y 90"_.
- Longitud fuera de `[-180, 180]` → _"La longitud debe estar entre -180 y 180"_.
- Valores no numéricos → _"Debe ingresar un valor numérico"_.

### 8.4 Campos vacíos

- Cualquier campo requerido vacío → _"Este campo es obligatorio"_.

### 8.5 Cancelar operación

- En el modal de baja, **Cancelar** cierra el modal sin cambios.
- En el formulario, **Cancelar** descarta cambios y vuelve al listado.

### 8.6 Error del servidor

- Cualquier fallo en backend → toast de error: _"Ocurrió un error. Inténtalo nuevamente."_.

---

## 9. Reglas de negocio

1. El **ID técnico** debe ser único en toda la base de datos (incluyendo dispositivos dados de baja).
2. La **baja es lógica**: nunca se elimina físicamente el registro; se marca como `decommissioned`.
3. Los dispositivos en estado `decommissioned` **no aparecen** en el mapa de calor ni en el dashboard de monitoreo.
4. Los dispositivos editados conservan su `id` y `created_at`; solo se actualiza `updated_at`.
5. Las **coordenadas se almacenan en grados decimales** con precisión de 6 decimales.
6. La validación geográfica es global: lat ∈ [-90, 90], long ∈ [-180, 180]. No se restringe a una región.
7. La sincronización con mapa y dashboard ocurre **al refrescar** dichas vistas (no en tiempo real).

## 11. Feedback al usuario

- ✅ **Éxito**: toast verde con mensaje confirmatorio.
- ❌ **Error de validación**: mensaje inline bajo el campo afectado.
- ⚠️ **Error de servidor**: toast rojo con mensaje genérico.
- ❓ **Confirmación destructiva**: modal con botones **Confirmar** / **Cancelar**.
